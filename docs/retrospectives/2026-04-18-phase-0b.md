## 주요 변경

### LSP 클라이언트 (핵심)
- `rootUri`: `None` → `file://` URI from project cwd.
- `capabilities`: 빈 `{}` → `textDocument.{synchronization, documentSymbol hierarchical, references, definition, rename prepareSupport}` + `workspace.workspaceEdit.documentChanges=True`.
- `_open_files` 버전 맵: `dict[str, int]` 도입, `_ensure_open(path)` 메서드가 `definition/references/rename` 직전에 didOpen 전송, 중복 방지.
- `_apply_workspace_edit`: `documentChanges` 우선, `changes` 폴백. `safe_open_fd` 경유로 심볼릭 링크 TOCTOU 방어.
- 타임아웃 테이블: pyright/ts/gopls 10s, clangd 15s, rust-analyzer 30s, jdtls 45s, default 15s.
- Shutdown 시퀀스: `shutdown` 요청 → 응답 수신 → `exit` 알림 → 2초 SIGTERM 유예 → SIGKILL.

### ToolRegistry 메타데이터
- `ToolCategory` enum: READ/WRITE/DESTRUCTIVE/SYMBOLIC/META.
- `ToolSideEffect` enum: NONE/FILESYSTEM/PROCESS/NETWORK/EXTERNAL.
- `BaseTool.tags: frozenset[str]`, `requires_confirmation: bool`.
- `ToolRegistry.filter()` 5 파라미터(category/side_effects/tags_any/tags_all/read_only) 다중 조건 AND.
- 17 기존 도구 전부 분류. `lsp_rename`의 `"symbol"` 태그 누락은 스펙 해석 결과로 기록(roadmap 재확인 필요 — 현재는 `{"lsp","refactor"}`).
- `is_read_only`를 property로 전환, `McpToolAdapter`의 인스턴스 대입 제거.

### CI 매트릭스
- `.github/workflows/lsp_tests.yml`: ubuntu-22.04 × python 3.11/3.12.
- pyright + typescript-language-server npm 설치.

- placeholder 2개로 CI 프리훅 검증.

## 예상 못한 이슈

### 1. `lsp_rename` 태그 스펙 드리프트
로드맵 원안: `{"lsp","symbol","refactor"}`. 실제 구현: `{"lsp","refactor"}`.
이유: 메타데이터 테스트의 `filter(tags_all={"lsp","symbol"})`가 정확히 2개(goto_definition, find_references)만 반환해야 했고, rename에 symbol 태그를 넣으면 3개가 됨. 계획 스펙과 테스트 스펙 간 충돌.

처리: 현 구현 유지. Phase F에서 모드/컨텍스트 프로필 설계 시 "refactor" 태그를 symbol 계열로 승격할지 재검토.

### 2. LSP 테스트 환경 의존
로컬에 pyright/typescript-language-server가 없어 placeholder 2개는 `xfailed`로 관찰됨. CI에서만 실제 실행. 정상 동작.

### 3. Phase 0A baseline과의 호환성
`check_import_graph.py` baseline 모드가 그대로 작동. 67 modules, 8 cycles (Phase 0A의 8과 동일). 분해/하드닝 과정에서 신규 순환 도입 0건.
